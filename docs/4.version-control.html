Conclusions --- class: center, middle # Motivations --- # Motivations There are a variety of motivations for developers to use version control systems: - Archiving - Synchronizing - Repair - Accountability - Experimentation --- # Motivations ## Archiving The need to clearly see the state of the project at a given point in time, and to compare versions. --- # Motivations ## Synchronizing The need for multiple developers to make sure they are all working off of the same version of a project's codebase. --- # Motivations ## Repair The need to sometimes undo changes that have been made and 'roll back' to an earlier version. --- # Motivations ## Accountability The need to see who did what and when they did it. --- # Motivations ## Experimentation The need to allow developers to try out new directions or ideas without concern about permanent damage if they fail. --- class: center, middle # Repositoriees --- # Repositories A repository (or 'repo') is the database where files are stored and tracked by the version control system. A repository can either be: - Local - Remote --- # Repositories ## Local A local repository resides on a developer's own machine. It is typically used to track their own work. --- # Repositories ## Remote A remote repository resides on some other machine... this could be another developer's machine, or somewhere on a shared server. --- class: center, middle # Trunk --- # Trunk The trunk or 'main line' of a repository is the authoritative best version of the code at any given point in time. --- # Trunk ## Benefits Maintaining code in a centralized trunk allows all developers to know where to find the best most authoritative version of the code at any given time. --- # Trunk ## Drawbacks If developers work directly on the code in the trunk (known as 'trunk-based development') without extra caution, bugs may appear in the trunk code. --- class: center, middle # Branches --- # Branches Branches represent variants of the code at any given point in time time. They are departures from the main line. --- # Branches ## Benefits Those wary of working directly on the trunk often make changes in isolated branches, separate from the trunk, and then 'merge' the changes into the code in the trunk once they are sure they have no bugs. --- # Branches ## Drawbacks If multiple developers work independently in separate branches for long periods of time, they may end up in 'merge hell', where the trunk has changed drastically since the time when they started the offshoot branch. --- class: center, middle # File vs. Repository Versioning --- # File vs. Repository Versioning When a change is made to a file, a new version number is issued. - In *file versioning systems*, a new version number is issued only for the file that has been changed. - In *repository versioning systems*, a new version number is issued for the repository as a whole, even when only one file has changed. --- # File vs. Repository Versioning ## File Versioning Each file in the repository has its own version number - so frequently changed files will typically have higher version numbers than stable files. ### Benefit - Easy to find the history of all versions of a given file. ### Drawback - Possibly difficult to determine all the individual files that comprise one logical release version. --- # File vs. Repository Versioning ## Repository Versioning Any changes trigger a new version number of the entire repository. ### Benefit - Easy to find all the individual files comprising a given system release version. ### Drawback - Possibly difficulty to determine the change history of one individual file. --- class: center, middle # Centralized vs. Distributed Versioning --- # Centralized vs. Decentralized Versioning Version control systems can either allow for centralized or decentralized control: - In *centralized version control systems*, one repository maintains the single authoritive copy of the code at all times. - In *decentralized version control systems*, all developers maintain and share the code in their own local repositories, with no single central authority. --- # Centralized vs. Distributed Systems ## Centralized Systems ### Benefits - Avoid conflicts by usually requiring developers to "check out" a file to change it and then "check in" the file when done. ### Drawbacks - Doesn't allow individual developers to track their own experiments before sharing them --- # Centralized vs. Distributed Systems ## Distributed Systems ### Benefits - Allows each developer to track their own work with all the features of version control on their local machines. - Can be used in a centralized version control workflow, but doesn't require it. ### Drawbacks - Without a careful workflow, developers may fall out of sync with one-another, leading to 'mergee hell'. --- class: center, middle # Conclusions Thank you.
